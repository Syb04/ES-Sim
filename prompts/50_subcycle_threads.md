# 50: イオンサブサイクリング + 粒子チャンク並列 (高速化)

## バックエンド設計 (実装済み — 本体モデルが直接実装。以下は記録)

### イオンサブサイクリング (`pic.ion_subcycle: int = 1`)

- イオンを N ステップに1回だけ、実効刻み N·dt で押す (最初のプッシュは step 0)。
  一定電界ではリープフロッグが厳密なので N によらず軌道が一致する
- 休止ステップ中はイオンが動かないため、電荷堆積ベクトルをキャッシュして再利用
  (`_f_ion_cache`)。イオン配列が変わる箇所 (プッシュ・MCC電離生成・イオン注入)
  で必ず無効化する
- イオンの MCC 衝突・初期/注入の半ステップ後退キックも N·dt で整合させる
- KE 診断は休止中は直近プッシュ時の値を流用 (`_last_ke_i`)
- N=1 (既定) は従来経路とビット単位で一致

### 粒子チャンク並列 (`pic.threads: int = 1`)

- walk 探索を粒子チャンク (threads 分割) に分けて ThreadPoolExecutor で並列実行。
  全種のチャンクをまとめて投入し種横断で負荷分散する
- 各粒子の walk は独立・決定的なので、結果は逐次実行と**ビット単位で一致**する
  (threads を変えても結果は不変)
- 4096 粒子未満の種はスレッド起動コストの方が高いため逐次フォールバック
- 効果はコア数と粒子数に依存 (2コア環境の実測: 40k粒子 CCP で
  sub=10 + threads=2 により 17.5 → 9.0 ms/step ≈ 1.9×)

ベンチマーク: `python backend/scripts/bench_pic.py --n-macro 40000 --sub 10 --threads 8`

## フロントエンド作業 (サブエージェント向け指示)

1. **types.ts**: `PicSettings` に `ion_subcycle?: number`(既定 1)と
   `threads?: number`(既定 1)を追加 (optional、コメントは日本語)

2. **PicPanel.tsx**: 「積分設定」系のセクション (n_macro / dt / n_steps などの近く) に
   2つの入力を追加:
   - 「イオンサブサイクル」: 整数 ≥1。ヒント文言例:
     「イオンを N ステップに1回 N·dt で押す (1=無効)。5〜10 で高速化、結果は近似」
   - 「スレッド数」: 整数 ≥1。ヒント文言例:
     「walk 探索の並列スレッド数。結果は1と完全一致。CPUコア数程度まで」
   - 値は `onChange({ ...pic, ion_subcycle: v })` の形で更新。
     入力には既存の `CommitNumberInput` を使い、Math.max(1, Math.round(v)) で整数化

3. 既存挙動を変えない (未指定の保存済みプロジェクトはバックエンド既定 1 で動く)
